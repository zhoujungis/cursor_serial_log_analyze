4) 后续排查顺序

1. 先排查 OOM：确认 LTWP.0、ipc 的职责，采集 OOM 前后的内存水位、进程 RSS/PSS、slab、ion/cma、视频/NPU/Agora 内存占用，判断是内存泄漏、峰值过高还是启动内存预算不足。

2. 核对是否发生反复重启：围绕多次 [0.xxx] 内核启动时间戳，查找 boot banner、watchdog、panic、reboot reason、电源复位原因，确认是整机重启、应用重启，还是日志文件拼接了多段启动。

3. 将 WiFi/BLE 断连与重启/OOM 时间线对齐：判断断连是 OOM/重启的结果，还是网络模块异常导致业务重启；重点查看 disconnect wifi 前后的调用来源、状态机、错误码、RSSI/AP 状态和绑定流程动作。

4. 排查存储与配置区：针对 read_meta_sector head error 0xffffffff、gz_sysctrl_read_iq_type crc error，检查分区表、meta/IQ 数据烧录、版本兼容、CRC 生成逻辑、掉电保护，以及是否存在读错分区或空白分区。

5. 排查 AV/ISP 初始化链路：结合 set clk_core_isp rate error、gz_hal_video_set_inversion error 1、IQ CRC 错误，确认 sensor/ISP 驱动、clock tree、DTS 配置、启动脚本写 sysfs 参数是否一致。

6. 定位 sh: write error: Invalid argument 的具体写入命令：在启动脚本中打开命令回显或增加节点名日志，找出哪个 sysfs/procfs 参数非法。

7. 排查 Cover cloud 参数来源：确认参数应由绑定流程、云端下发还是本地持久化提供；检查参数未设置是否与 WiFi 断连、meta 读取失败、进程重启有关。

8. 降低日志噪声并复测：AGORA init_all_log_mod、EROF DAX warning 等先作为低优先级噪声处理，复测时保留关键模块日志，避免高频重复日志掩盖 OOM/断连前因。
